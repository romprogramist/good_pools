# Form source tracking — design

**Дата:** 2026-05-12
**Контекст:** Сайт good-pools.ru. Сейчас в письмах-заявках источник формы показан мелким текстом в подзаголовке («Услуга · 12.05.2026»), а в БД хранится только короткий код (`source`). При просмотре письма с телефона неясно, какая именно форма прислала заявку, а для аналитики «с какой страницы пришла заявка» данных вообще нет.

## Цель

1. В письме сразу видно, **с какой формы** заявка (крупный баннер сверху).
2. В БД есть **человеко-читаемое название формы** для прямых SQL-запросов (без нужды переводить коды в коде).
3. В БД фиксируется **URL-путь страницы**, с которой пришла заявка.

Решение охватывает 5 форм: `service`, `ask`, `consult`, `quiz`, `interest-popup`.

## Изменения схемы БД

Новая миграция `db/migrations/0004_leads_source_label_and_page.sql`:

```sql
ALTER TABLE leads ADD COLUMN IF NOT EXISTS source_label VARCHAR(64);
ALTER TABLE leads ADD COLUMN IF NOT EXISTS page_path    VARCHAR(255);
```

- `source_label` — денормализованная человеко-читаемая копия источника, **пишется при INSERT** значением из `SOURCE_LABEL` в `lib/mailer.js`. Если позже русское название поменяется в коде, старые записи сохранят исторически верное значение. Это удобнее, чем JOIN/CASE в каждом SELECT.
- `page_path` — URL-путь страницы, где была подана заявка (например `/index.html`, `/portfolio.html`). Без query и hash.
- Обе колонки NULLable: миграция применится мгновенно даже на большой таблице, старые записи останутся валидными.

`db/init.sql` обновляется параллельно так, чтобы fresh install сразу создавал эти колонки (init.sql + миграции должны сходиться — см. CLAUDE.md проекта).

Индексы не добавляются: основной запрос по-прежнему по `source` (индекс уже есть). `source_label` дублирует `source` для удобства чтения, а не для фильтрации. `page_path` может пригодиться для аналитики — отдельным запросом, индекс пока не нужен.

## Backend

### `lib/mailer.js`

**`SOURCE_LABEL` — обновлённые значения** (поправляем размытые формулировки на однозначные):

```js
const SOURCE_LABEL = {
  service:           'Сервисное обслуживание',
  ask:               'Задать вопрос',
  consult:           'Заказ консультации',
  quiz:              'Подбор бассейна (квиз)',
  'interest-popup':  'Интерес к модели (поп-ап)'
};
```

**`buildEmail(lead)` — изменения шаблона:**

- **Новый баннер сверху письма** (до существующей синей шапки): жёлтый/янтарный фон (`#fef3c7`, текст `#92400e` — высокий контраст, «warning»-семантика), крупный текст: «**ФОРМА: Сервисное обслуживание**», шрифт ~18px bold. Inline-стили — email-клиенты не понимают внешний CSS.
- В существующей синей шапке: дата остаётся, имя источника убирается (его теперь видно в баннере).
- В секции «Контакт» появляется строка «Страница: /portfolio.html» — только если `lead.page_path` непустой.
- Plain-text часть синхронизируется со структурой HTML: первая строка `ФОРМА: Сервисное обслуживание`, далее `Страница: /portfolio.html` (если есть).

Fallback: если `lead.source_label` пустой (например, сообщение приходит из старой записи или из тестов без бэкенда), mailer берёт `SOURCE_LABEL[lead.source]` как запасной вариант. Письмо никогда не должно показывать сырой код `service`/`ask` без перевода.

### `routes/api.js` → POST `/api/leads`

- Принимаем новое **top-level** поле `page` в теле запроса (не в `payload`). Нормализация:
  - если `page` не строка или пустое — `page_path = NULL`;
  - если строка длиннее 255 — отрезаем до 255 (лид важнее точного URL, заявку не отклоняем).
- При INSERT добавляются две новые колонки:
  - `source_label` = `SOURCE_LABEL[source]` (берётся на бэке, не из тела запроса — клиент не должен влиять на отображаемое название).
  - `page_path` = нормализованный `page`.
- В вызов `mailer.sendLeadEmail({ ... })` пробрасываются `source_label` и `page_path`.

## Frontend (5 форм)

В тело POST-запроса `/api/leads` каждая форма добавляет одно поле:

```js
page: window.location.pathname
```

Файлы:
- `js/service.js` — форма сервисного обслуживания (`index.html`)
- `js/ask.js` — «Задать вопрос» (`index.html`)
- `js/consult.js` — модалка консультации (открывается с любой страницы)
- `js/quiz.js` — модалка квиза (любая страница)
- `js/interest-popup.js` — поп-ап на страницах моделей

Почему только `pathname` (без query/hash):
- query на этом сайте маркетингово не используется
- hash — внутренние якоря, не несут смысла источника
- VARCHAR(255) перекрывает самый длинный путь с запасом

## Поток данных (end-to-end)

```
[пользователь на /portfolio.html заполняет consult-модалку]
       │
       ▼
[js/consult.js → POST /api/leads с { source: 'consult', ..., page: '/portfolio.html' }]
       │
       ▼
[routes/api.js: валидация → INSERT leads (source='consult', source_label='Заказ консультации', page_path='/portfolio.html', ...)]
       │
       ▼
[mailer.sendLeadEmail с теми же полями]
       │
       ▼
[admin email]
   ┌──────────────────────────────────┐
   │ ФОРМА: Заказ консультации        │ ← баннер
   ├──────────────────────────────────┤
   │ Новая заявка с сайта             │
   │ 12.05.2026, 22:50                │ ← синяя шапка (без имени источника)
   ├──────────────────────────────────┤
   │ Контакт                          │
   │ Имя: ...                         │
   │ Телефон: ...                     │
   │ ...                              │
   │ Страница: /portfolio.html        │ ← новая строка
   └──────────────────────────────────┘
```

## Edge cases и совместимость

- **Старый кэш JS у клиента, не передаёт `page`.** API ляжет `NULL`, в письме строки «Страница» не будет. Не падаем.
- **Старая запись в БД без `source_label`.** Mailer берёт fallback из `SOURCE_LABEL[source]`. Если и source неизвестен — показываем сам код в баннере.
- **Длинный URL.** Обрезаем до 255. Лид важнее точности path.
- **Миграция повторно.** `IF NOT EXISTS` — идемпотентна.
- **Откат.** Если что-то ломается на проде — откатывать нечего: колонки nullable, новый код mailer'а работает и без них (через fallback). Можно просто revert PR.

## Тесты

Автотестов в проекте нет. Проверка — manual:

1. **Локальная превью письма.** Скрипт `.tmp-mailer-preview.js` (как в прошлой итерации) рендерит 2–3 кейса с заполненными `source_label` и `page_path`, сохраняет HTML, открываем в браузере — визуально проверяем баннер и строку «Страница».
2. **POST `/api/leads` через curl** (после деплоя):
   ```
   curl -X POST https://good-pools.ru/api/leads \
     -H 'Content-Type: application/json' \
     -d '{"source":"ask","name":"Test","phone":"+79000000000","consent":true,"page":"/portfolio.html","payload":{"question":"тест"}}'
   ```
   Письмо должно прийти с новым баннером и строкой «Страница: /portfolio.html».
3. **Проверка БД** (через SSH-туннель):
   ```
   SELECT id, source, source_label, page_path, created_at
   FROM leads
   ORDER BY id DESC LIMIT 5;
   ```
   Новые записи должны содержать `source_label` и `page_path`.

## Деплой

Стандартный flow: `git push origin main` → GitHub Actions → rsync → `npm install --omit=dev` → `node db/migrate.js` → `pm2 restart good-pools`. Миграция накатится сама, рестарт подхватит новый код.

## Out of scope

- Админ-страница «Заявки» с фильтром по форме/странице — отдельный спек, если решим делать.
- Аналитика по конверсии форм/страниц — данные теперь в БД, но дашборд не строим.
- Передача UTM-меток / referer — query/hash не сохраняем по решению выше.
